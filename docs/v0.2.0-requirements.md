# wiki-cli v0.2.0 需求文档

## 版本目标

将 wiki-cli 从"手动触发的一次性编译工具"升级为"harness 原则驱动的三级知识管理系统"。

## 核心原则（Harness 四维度）

### ① 存在成本 vs 触发收益

- **被动存在成本**：系统在无人操作时消耗的资源（存储、计算、运维）
- **主动执行成本**：执行一次操作的成本（LLM 调用、IO、人工时间）
- **规则**：被动存在成本必须趋近于零；主动执行成本必须与收益成正比

### ② 反馈闭环度

| 级别 | 定义 | 适用场景 |
|------|------|----------|
| 开环 | 无反馈 | Capture（无需反馈） |
| 半闭环 | 反馈来自独立信号 | Compile（拒绝信号回灌） |
| 全闭环 | 反馈来自独立执行者 | Promote（人工确认） |

**核心判定标准**：反馈来源是否独立于执行者。

### ③ 触发确定性

| 级别 | 说明 | 本系统使用 |
|------|------|-----------|
| 确定性 | 条件满足必然触发 | Capture(#wiki)、Compile(guard)、GC(手动) |
| 条件性 | 满足条件后概率触发 | — |
| 概率性 | 可能触发也可能不触发 | — |
| 祈祷式 | 完全不可控 | — |

### ④ 层次正确性

参考 Anthropic 五层架构，每一层做好自己的事：

- **编译层**只做编译，不做质量判断
- **审计层**只做质量判断，不做编译
- **工具层**(CLI)不实现本应由人工层承担的决策

## 三级模型：Capture → Compile → Promote

### 第一级：Capture（采集）

**模型**：标签驱动的"黑洞"

**触发方式**：
1. **主路径**：OpenClaw Plugin — 监听所有经过 OpenClaw 的对话，命中 `#wiki` 关键字时自动提取上下文
2. **兜底路径**：CLI `wiki capture` — 手动输入文本/文件/stdin

**行为**：
- 提取标记消息 + 前后 N 条上下文（默认 N=3）
- 生成 .md 文件写入 `raw/notes/`
- 存入成本 = 零（只需标记，不需要复制粘贴或运行 CLI）

**Harness 属性**：
- 触发：确定性（#wiki 是明确的触发条件）
- 反馈：开环（无需反馈，采集本身不产生质量判断）
- 存在成本：零（常驻进程成本由 OpenClaw 承担，非 wiki-cli 的存在成本）

### 第二级：Compile（编译）

**模型**：静默的"堆肥机"

**触发方式**：`wiki compile` 命令，配合 cron 或手动触发
- **Guard 机制**：扫描 raw/ 无未处理文件 → 直接退出，零 LLM 调用
- 不使用裸 cron（时间驱动），必须配合 guard（事件驱动）

**行为**：
- 扫描 raw/ 中未处理文件
- 调用低成本 LLM 编译为结构化草稿
- 输出到 `compiled/YYYY/MM/`，不触碰 wiki/
- 注入 `compile_feedback.md` 中的反馈信号

**反馈闭环**：
- Promote 阶段的拒绝原因写入 `compile_feedback.md`
- 下次 Compile 时读取并注入 LLM prompt
- 这是半闭环：反馈来自 Promote（独立于 Compile 的执行者），但不会自动调整 Compile 参数

**Harness 属性**：
- 触发：确定性（guard 保障，无新文件不执行）
- 反馈：半闭环（拒绝信号回灌）
- 存在成本：低（仅处理增量，不重复编译）

### 第三级：Promote（审计晋升）

**模型**：极简的"点兵点将"

**触发方式**：
1. 手动 `wiki promote` 审核待晋升列表
2. 周期性推送提醒（由外部调度，CLI 本身不实现定时推送）

**行为**：
- 列出 `compiled/` 中 `status: "pending_audit"` 的文件
- 按引用频率排序（频率仅做排序，不做筛选闸门）
- 用户确认 → 移入 wiki/（晋升）
- 用户拒绝 → 写入原因到 compile_feedback.md，删除 compiled 文件

**频率 = 排序 ≠ 筛选**：
- 一个反复出现的 bug 报告可能被提 10 次，但一条关键架构决策只出现 1 次
- 频率只决定"先看哪个"，真正的闸门是人工确认

**Harness 属性**：
- 触发：条件性（需要人工介入）
- 反馈：全闭环（人工确认 + 拒绝信号回灌 Compile）
- 存在成本：低（仅展示待审核列表）

## 数据生命周期

### raw/ 文件生命周期

```
raw/notes/note.md
  │
  ├─ Compile 选中 → 编译成 compiled/2026/04/note.md
  │                   raw 原文件保留作为源数据
  │
  └─ 90 天未被 Compile 选中
      → 移入 raw/archive/
          │
          └─ 再过 90 天 → 删除
```

### compiled/ 文件生命周期

```
compiled/2026/04/topic.md
  │
  ├─ Promote 确认 → 内容移入 wiki/，compiled 文件删除
  │
  ├─ Promote 拒绝 → 原因写入 compile_feedback.md，compiled 文件删除
  │
  └─ 180 天未审核 → auto-reject，写入 compile_feedback.md
```

---

## 约束条件

1. **raw/ 不可变**：Capture 写入后，Compile 和 Promote 不修改、不删除 raw 原文件（仅 GC 可归档/删除）
2. **compiled/ 是暂存区**：不承载长期数据，晋升或拒绝后清除
3. **wiki/ 需人工确认**：任何进入 wiki/ 的内容必须经过 Promote 确认
4. **compile_feedback.md 人类可读**：不是内部状态文件，用户可以直接查看和编辑
5. **向后兼容**：`wiki ingest` 保留为 alias，打印 deprecation warning
6. **OpenClaw 是唯一 chokepoint**：Capture Plugin 部署在 OpenClaw 上，覆盖所有前端渠道
